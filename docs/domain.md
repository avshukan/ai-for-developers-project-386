# Domain Model

This document defines the domain model and business rules for the Call Booking MVP.

It must be consistent with:

* `docs/glossary.md`
* `docs/product.md`
* `api/main.tsp`

## Domain Summary

The application allows one predefined host to create event types, publish available time, and receive bookings from guests.

Guests do not have accounts.
Guests choose an event type, see available slots for the next 14 days, and book one available slot with name and email.

## Roles

### Host

The host is the calendar owner role.

For MVP:

* there is only one predefined host
* host is not authenticated
* host pages are public
* host is not necessarily stored as a database entity

The host can:

* create event types
* publish availability
* view upcoming bookings

### Guest

The guest is the person booking a meeting.

For MVP:

* guest has no account
* guest is represented only by name and email inside a booking

The guest can:

* view event types
* choose an event type
* view available slots
* create a booking

## Entities

### Event Type

An event type describes a bookable meeting type.

Fields:

* `id`
* `title`
* `description`
* `durationMinutes`

Rules:

* `id` is generated by the system.
* `title` is required.
* `description` is required.
* `durationMinutes` is required.
* `durationMinutes` must be positive.
* For MVP, event type duration is fixed at 30 minutes: the API accepts only `30` and rejects any other value with `400 validation_error`. The general "must be positive" rule applies once configurable duration is introduced (see open question).
* Event type is created by the host.
* Guest must select an event type before viewing slots.

Open question:

* Whether configurable event type duration is needed after the first MVP: `TBD`.

### Availability

Availability is a time range published by the host.

Fields:

* `id`
* `startAt`
* `endAt`
* `createdAt`

Rules:

* `id` is generated by the system.
* `startAt` is required.
* `endAt` is required.
* `endAt` must be after `startAt`.
* Availability is used to derive slots.
* Availability is not a booking.
* Availability does not reserve time by itself.
* Recurring availability is out of scope.
* Editing and deleting availability are out of scope for MVP.

### Slot

A slot is a fixed 30-minute bookable time interval derived from availability.

Fields:

* `startAt`
* `endAt`
* `status`

Rules:

* Slot is derived, not necessarily stored.
* Slot duration is fixed at 30 minutes.
* `status` is `available` or `booked`.
* Only full 30-minute slots inside availability are returned.
* Slot must not start in the past.
* Guest sees available slots for the next 14 days.
* Booked time must not be returned as available.
* Slots are global; the event type is selected for the booking, not the slot.
* Time conflicts are global across all event types.

### Booking

A booking is a confirmed reservation of time by a guest.

Fields:

* `id`
* `eventTypeId`
* `startAt`
* `endAt`
* `guestName`
* `guestEmail`
* `createdAt`

Rules:

* `id` is generated by the system.
* `eventTypeId` is required.
* `startAt` is required.
* `endAt` is `startAt` plus 30 minutes.
* `guestName` is required.
* `guestEmail` is required.
* `guestEmail` must pass simple MVP email validation.
* Booking must reference an existing event type.
* Booking must use an available slot.
* Booking must not be created in the past.
* Booking must not overlap with any existing booking.
* Double booking must be prevented on the backend.

## Core Invariants

These rules must always hold.

### No Double Booking

Two bookings cannot overlap in time.

This applies even when bookings use different event types.

Example:

```text
Booking A:
10:00–10:30, Event Type: Intro Call

Booking B:
10:00–10:30, Event Type: Support Call

Result:
Booking B must be rejected.
```

Overlap is checked by time interval:

```text
new.startAt < existing.endAt
AND
new.endAt > existing.startAt
```

### No Past Booking

A booking cannot be created if its slot start time is earlier than current server time.

### Booked Time Is Not Available

If a booking exists for a time interval, that interval must not be returned as available for any event type.

### API Contract Must Follow Domain Rules

Every API behavior must trace to:

* a product scenario in `docs/product.md`
* a domain rule in this file
* or an ADR

## Time Rules

* API date-time values use ISO 8601.
* API date-time values are represented in UTC.
* Server time is the source of truth.
* UI may display dates in browser local time.
* Slot search window is the next 14 days from current server time.
* Past slot means `slot.startAt < current server time`.
* Upcoming booking means `booking.startAt >= current server time`.

UI date/time display format: `TBD`.

## Slot Generation Rules

Input:

* fixed 30-minute slot length
* availability ranges
* existing bookings
* current server time

Process:

1. Take availability ranges within the next 14 days.
2. Split each availability range into full 30-minute intervals.
3. Ignore intervals that start in the past.
4. Ignore intervals that overlap existing bookings.
5. Return remaining intervals as available slots.

Example:

```text
Availability:
10:00–12:00

Slot length:
30 minutes

Generated slots:
10:00–10:30
10:30–11:00
11:00–11:30
11:30–12:00
```

If availability length is not divisible by the slot length, the leftover part is ignored.

Example:

```text
Availability:
10:00–11:20

Slot length:
30 minutes

Generated slots:
10:00–10:30
10:30–11:00

Ignored:
11:00–11:20
```

## Validation Rules

### Event Type Validation

Invalid when:

* title is empty
* description is empty
* durationMinutes is missing
* durationMinutes is not 30 (MVP fixed duration; the general rule is durationMinutes is less than or equal to zero once configurable duration is introduced)

### Availability Validation

Invalid when:

* startAt is missing
* endAt is missing
* endAt is not after startAt

### Booking Validation

Invalid when:

* eventTypeId is missing
* startAt is missing
* guestName is empty
* guestEmail is empty
* guestEmail has invalid MVP format
* selected slot is in the past
* selected slot is outside availability

Two booking failures are deliberately **not** `400 validation_error`:

* A booking that references a non-existent `eventTypeId` has a valid request
  shape but a missing resource, so the API returns `404 not_found`.
* A booking whose slot overlaps an existing booking is a valid request that
  loses a race for the slot, so the API returns `409 booking_conflict`.

## Public Pages Security Limitation

Host pages are public in MVP.

This means anyone who knows the URL can:

* create event types
* publish availability
* view upcoming bookings

This is intentional for the learning project.
It is not a production security model.

Authentication must not be added unless product scope changes through ADR.

## Domain Scenarios

### Host Creates Event Type

1. Host submits title, description, and duration.
2. System validates input.
3. System creates event type.
4. Event type becomes visible to guests.

### Host Publishes Availability

1. Host submits startAt and endAt.
2. System validates input.
3. System saves availability.
4. Availability can be used for slot generation.

### Guest Views Event Types

1. Guest opens booking page.
2. System returns available event types.
3. Guest selects one event type.

### Guest Views Slots

1. Guest selects event type.
2. System calculates slots for the next 14 days.
3. System removes past slots.
4. System removes booked intervals.
5. Guest sees available slots.

### Guest Creates Booking

1. Guest selects event type and slot.
2. Guest enters name and email.
3. System validates input.
4. System checks slot availability.
5. System checks global booking conflict.
6. System creates booking.
7. Selected time becomes unavailable.

### Host Views Bookings

1. Host opens bookings page.
2. System returns upcoming bookings.
3. Bookings are shown across all event types.

## Persistence Notes

For the MVP, storage is an in-memory store that resets on restart
(`docs/adr/0002-backend-stack-and-storage.md`). A durable database remains a
future decision.

The storage layer must support:

* event types
* availability ranges
* bookings
* atomic prevention of double booking

Atomic double-booking prevention is required for correctness.

This may require a database-level constraint or transaction depending on storage choice.
